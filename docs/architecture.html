`schema.md`

`schema.md` tells the LLM how to behave. It defines directory structure, page format, frontmatter conventions, cross-linking rules, workflows, and naming conventions.

Because the agent reads `schema.md` at the start of every operation, editing it is the main way to customize behavior for different domains or wiki structures.

    ├── config.py
    └── templates.py

Module roles

  • cli.py: Typer commands such as `init`, `ingest`, `query`, `lint`, `status`, and `config`.
  • agent.py: Provider backends, agent loop, and prompt builders.
  • tools.py: File tools used by the LLM while operating on the wiki.
  • config.py: API key and provider resolution logic.
  • templates.py: Default contents written during `wiki init`.

LLM tools

Tool Purpose
read_file(path) Reads files relative to the wiki root.
write_file(path, content) Writes or overwrites a file and creates parent directories as needed.
append_to_file(path, content) Appends content, typically used for `log.md`.
list_directory(path) Lists files and subdirectories.
search_wiki(query) Searches markdown files in the wiki for relevant terms.

Workflow tips from the README

  • Use Obsidian as a frontend if you want graph view and plugins like Dataview or Marp.
  • Ingest one source at a time for better results.
  • Use `--save` on valuable queries so good answers become reusable knowledge.
  • Version the wiki with git so ingests and query saves become natural commits.

Current limitations

  • Native ingestion focuses on plain text-like source formats.
  • The README suggests the current index-based approach works best up to roughly 100 to 200 sources.
  • Images are not automatically processed from sources.
  • Costs vary based on model, source size, and wiki size.
